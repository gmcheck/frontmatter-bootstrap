```text id="jlwm_final"
开发一个 Obsidian 插件：

frontmatter-bootstrap

========================================
一、插件定位（核心）
========================================

这是一个：

"长期主义 frontmatter 基础设施插件"

插件目标：

1. 新建 Markdown 笔记时：
自动插入统一 YAML metadata 模板

2. （暂时不实现这部分功能）自动初始化 Obsidian Property Schema：
让固定字段具备：
- Select
- Date
- Tags

Obsidian 暂时对这部分支持能力有限，这部分等 Properties UI 能力。

3. 提供：
轻量级 Schema Validation（手动健康检查）

========================================
二、核心设计哲学（必须遵守）
========================================

插件必须：

- 极简
- 长期稳定
- 低侵入
- 不做 AI
- 不自动分类
- 不自动生成标签
- 不分析正文
- 不后台扫描
- 不实时校验

----------------------------------------

插件目标：

不是：
"智能知识系统"

而是：

"长期稳定 metadata 基础设施"

========================================
三、Metadata Bootstrap 功能
========================================

当用户创建新的 Markdown 文件时：

如果：
- 文件为空
- 没有 frontmatter

则：

自动插入：

---
type: note
domain:
status: active
created: 2026-05-09
updated: 2026-05-09
source: personal
tags:
---

----------------------------------------

日期格式：

YYYY-MM-DD

========================================
四、最终 Metadata Schema（固定）
========================================

----------------------------------------
1. type
----------------------------------------

字段类型：
Select

使用 Vocabulary 系统，用户可自定义允许值。

默认 vocabulary：
- note — 普通知识笔记
- project — 具有生命周期的项目型内容

默认：
note

----------------------------------------
2. domain
----------------------------------------

字段类型：
Select

默认值：

- music
- bigdata
- aigc
- ecommerce
- programming
- finance
- system

允许用户扩展。

插件不要自动推断 domain。

----------------------------------------
3. status
----------------------------------------

字段类型：
Select

使用 Vocabulary 系统，用户可自定义允许值。

默认 vocabulary：
- active — 当前活跃维护中
- done — 已完成，但仍保留在活跃知识空间
- archived — 已退出活跃知识空间

默认：
active

----------------------------------------
4. created
----------------------------------------

字段类型：
Date

格式：

YYYY-MM-DD

仅创建时写入一次。

----------------------------------------
5. updated
----------------------------------------

字段类型：
Date

格式：

YYYY-MM-DD

文件保存时自动更新。

----------------------------------------
6. source
----------------------------------------

字段类型：
Select

使用 Vocabulary 系统，用户可自定义允许值。

默认 vocabulary：
- personal — 以个人思考/整理为主
- external — 以外部资料整理为主

默认：
personal

----------------------------------------
7. tags
----------------------------------------

字段类型：
Tags

========================================
五、Vocabulary 系统
========================================

type、status、source 字段使用 Vocabulary 系统：

1. 用户可以在设置页面添加、删除、修改 vocabulary 值
2. 每个 vocabulary 值包含：value（值）和 description（描述）
3. 默认值从 vocabulary 中选择
4. 健康检查时，检查字段值是否在 vocabulary 中

设置页面 UI：

每个 vocabulary 编辑器显示为独立卡片：

┌─────────────────────────────────────────────────────┐
│ Type  Defines the structural role       [+ Add]    │
│                                                     │
│  note          General knowledge note            ✕  │
│  project       Project-oriented...               ✕  │
│                                                     │
│  Default type            [note ▼]                  │
└─────────────────────────────────────────────────────┘

- 值列表：显示当前所有 vocabulary 值及其描述
- 删除按钮：点击 ✕ 删除该值
- + Add 按钮：位于卡片右上角，点击添加新值
- Default 下拉框：从 vocabulary 中选择默认值

========================================
六、updated 自动更新时间
========================================

当 Markdown 文件保存时：

如果：
存在：

updated:

字段。

则：

自动更新：

updated: YYYY-MM-DD

----------------------------------------

要求：

1. 仅更新 updated
2. 不修改 created
3. 不修改其他字段
4. 不覆盖 frontmatter
5. 不修改正文

========================================
七、Schema Validation（最终版）
========================================

Validation：

是：

"Metadata Health Check"

而不是：

实时约束系统。

----------------------------------------

Validation：

仅允许：

用户手动执行。

========================================
八、Validation 功能（统一设计）
========================================

不要拆分：

- Validate Current Note
- Validate Folder
- Validate Vault

多个命令。

----------------------------------------

改为：

一个统一入口。

========================================
九、最终 Validation 命令（正式版）
========================================

Command Palette：

Metadata Health Check

========================================
十、Metadata Health Check 执行逻辑
========================================

执行命令后：

弹出选择菜单。

用户选择：

----------------------------------------
1. Current Note
----------------------------------------

检查当前 Markdown 文件。

----------------------------------------
2. Current Folder
----------------------------------------

检查当前目录。

----------------------------------------
3. Entire Vault
----------------------------------------

检查整个 Vault。

========================================
十一、Validation 检查内容
========================================

----------------------------------------
1. Required Field Validation
----------------------------------------

检查：

- type
- status
- created
- updated

是否存在。

----------------------------------------
2. Enum Validation
----------------------------------------

检查字段值是否在 vocabulary 中：

----------------------------------------
type
----------------------------------------

检查 type vocabulary。

----------------------------------------
status
----------------------------------------

检查 status vocabulary。

----------------------------------------
source
----------------------------------------

检查 source vocabulary。

----------------------------------------
domain
----------------------------------------

允许默认：

- music
- bigdata
- aigc
- ecommerce
- programming
- finance
- system

----------------------------------------
3. Date Format Validation
----------------------------------------

检查：

created
updated

是否符合：

YYYY-MM-DD

----------------------------------------
4. Unknown Field Validation（可选）
----------------------------------------

检查：

可能 typo 字段。

例如：

statu → status

========================================
十二、Validation 输出方式
========================================

不要：

- 实时弹窗
- 实时 Warning
- 后台 Validation
- 阻塞 UI

========================================
十三、Validation Report（正式版）
========================================

Validation 执行完成后：

生成：

Metadata Health Report

----------------------------------------

推荐形式：

Markdown Report

========================================
十四、Validation Report 示例
========================================

Metadata Health Report

File:
Knowledge/Music/Jazz.md

Warnings:

1. Missing required field:
status

2. Invalid enum value:
source: internet

Allowed:
personal, external

3. Invalid date format:
created: 2026/05/09

Expected:
YYYY-MM-DD

========================================
十五、Validation 严重级别
========================================

仅支持：

- warning
- info

----------------------------------------

不要：

Error Blocking

========================================
十六、技术实现要求
========================================

允许：

使用：

app.metadataTypeManager

初始化 Property Definitions。

----------------------------------------

要求：

1. 幂等执行
2. 不覆盖用户已有配置
3. 不破坏已有 schema

========================================
十七、Frontmatter 操作要求
========================================

必须使用：

processFrontMatter()

或：

安全 YAML API。

----------------------------------------

禁止：

正则暴力替换 frontmatter。

========================================
十八、项目结构（正式版）
========================================

src/
    main.ts
    settings.ts
    template.ts
    metadata.ts

    validation/
        engine.ts
        report.ts
        validators/
            required.ts
            enum.ts
            date.ts
            unknown.ts

    types.ts

========================================
十九、模块职责
========================================

main.ts

插件入口。

----------------------------------------

settings.ts

插件设置。

----------------------------------------

template.ts

metadata 模板生成。

----------------------------------------

metadata.ts

frontmatter 安全读写。

----------------------------------------

validation/engine.ts

Validation Engine。

----------------------------------------

validation/report.ts

生成 Metadata Health Report。

----------------------------------------

validators/*

各类 validation 检查。

========================================
二十、性能要求（非常重要）
========================================

插件必须：

1. 不全库后台扫描
2. 不实时 Validation
3. 不影响启动速度
4. 不增加大型依赖
5. 不监听无关事件

----------------------------------------

Validation：

仅在用户主动执行时运行。

========================================
二十一、明确禁止实现的功能
========================================

不要实现：

- AI 分类
- 自动 domain 推断
- 自动 tags
- embedding
- graph analysis
- OCR
- 云同步
- 向量数据库
- 自动摘要
- 实时 Validation
- 自动修复

========================================
二十二、最终设计目标
========================================

这个插件：

不是：

"智能 PKM 平台"

而是：

一个：

- 极简
- 稳定
- 长期主义
- 低维护
- 几乎无感存在

的 metadata 基础设施插件。

========================================
二十三、最终输出要求
========================================

请直接生成：

1. 完整插件代码
2. manifest.json
3. package.json
4. tsconfig.json
5. 所有 TypeScript 文件
6. README
7. 可直接 npm install / build 的完整工程

确保：

符合 Obsidian 社区插件规范，
并可以直接运行。
```
