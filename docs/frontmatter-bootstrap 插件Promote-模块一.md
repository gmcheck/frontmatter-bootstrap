```text
开发一个 Obsidian 插件。

插件名称：
frontmatter-bootstrap

插件目标：
当用户创建新的 Markdown 笔记时，
自动插入统一、稳定、极简的 YAML frontmatter metadata 模板。

插件定位：
这是一个"metadata 基础设施插件"，
不是 AI 插件，
不是知识管理系统，
不是自动分类工具。

核心哲学：
- 极简
- 长期稳定
- 低侵入
- 可预测
- 确定性
- 不做语义理解
- 不做智能分析

========================================
一、核心功能
========================================

当用户创建新的 Markdown 文件时：

如果文件为空：
自动插入 YAML metadata 模板。

如果文件已有 frontmatter：
插件不处理。

如果文件不是 Markdown：
插件不处理。

========================================
二、默认 metadata 模板
========================================

默认插入：

---
type: note
domain:
status: active
created: {{date}}
updated: {{date}}
source: personal
tags:
---

日期格式：

YYYY-MM-DD

例如：

---
type: note
domain:
status: active
created: 2026-05-09
updated: 2026-05-09
source: personal
tags:
---

========================================
三、字段设计说明（必须遵守）
========================================

type:
使用 Vocabulary 系统，用户可自定义允许值。

默认 vocabulary：
- note — 普通知识笔记
- project — 具有生命周期的项目型内容

默认值：
note

----------------------------

domain:
默认空值。

插件不要自动推断 domain。
不要分析目录。
不要 AI 分类。

用户自行填写。

----------------------------

status:
使用 Vocabulary 系统，用户可自定义允许值。

默认 vocabulary：
- active — 当前活跃维护中
- done — 已完成，但仍保留在活跃知识空间
- archived — 已退出活跃知识空间

默认值：
active

----------------------------

created:
创建时间。

格式：
YYYY-MM-DD

仅创建时写入一次。

以后绝对不允许修改。

----------------------------

updated:
最后更新时间。

格式：
YYYY-MM-DD

文件保存时自动更新。

----------------------------

source:
使用 Vocabulary 系统，用户可自定义允许值。

默认 vocabulary：
- personal — 以个人思考/整理为主
- external — 以外部资料整理为主

默认值：
personal

----------------------------

tags:
默认空数组。

========================================
四、Vocabulary 系统
========================================

type、status、source 字段使用 Vocabulary 系统：

1. 用户可以在设置页面添加、删除、修改 vocabulary 值
2. 每个 vocabulary 值包含：value（值）和 description（描述）
3. 默认值从 vocabulary 中选择
4. 健康检查时，检查字段值是否在 vocabulary 中

设置页面 UI：

每个 vocabulary 编辑器显示为独立卡片：

┌─────────────────────────────────────────────────────┐
│ Type  Defines the structural role       [+ Add]    │
│                                                     │
│  note          General knowledge note            ✕  │
│  project       Project-oriented...               ✕  │
│                                                     │
│  Default type            [note ▼]                  │
└─────────────────────────────────────────────────────┘

- 值列表：显示当前所有 vocabulary 值及其描述
- 删除按钮：点击 ✕ 删除该值
- + Add 按钮：位于卡片右上角，点击添加新值
- Default 下拉框：从 vocabulary 中选择默认值

========================================
五、updated 自动更新时间功能
========================================

功能要求：

当 Markdown 文件保存时：

如果文件包含 frontmatter：
并且存在：

updated:

字段。

则自动更新为当前日期。

格式：
YYYY-MM-DD

例如：

updated: 2026-05-09

----------------------------------------

重要规则：

1. 仅更新 updated
2. 不修改 created
3. 不修改其他 metadata
4. 不修改正文
5. 不覆盖用户自定义 frontmatter

========================================
六、安全原则（非常重要）
========================================

插件必须：

1. 不扫描历史文件
2. 不修改旧文件
3. 不分析正文
4. 不自动生成标签
5. 不自动分类
6. 不使用 AI
7. 不联网
8. 不做目录推断
9. 不覆盖已有 frontmatter
10. 不重复插入 metadata

========================================
七、用户设置项
========================================

插件设置页面允许配置：

----------------------------------------
Base Options
----------------------------------------

包含 type、status、source 三个 vocabulary 编辑器。

每个编辑器：
- 显示当前所有 vocabulary 值
- 支持添加新值（value + description）
- 支持删除值
- 支持选择默认值

----------------------------------------
功能开关
----------------------------------------

{
  "enableCreated": true,
  "enableUpdated": true,
  "autoUpdateModifiedDate": true
}

----------------------------------------
Domain 选项
----------------------------------------

{
  "domainVocabulary": ["music", "bigdata", "aigc", "ecommerce", "programming", "finance", "system"]
}

----------------------------------------
模板内容
----------------------------------------

允许用户自定义 YAML 模板。

支持变量：

{{date}}

替换为：

YYYY-MM-DD

========================================
八、事件监听逻辑
========================================

实现以下逻辑：

----------------------------------------
1. 新建文件监听
----------------------------------------

监听：

vault.on("create")

当创建 Markdown 文件时：

检查：

- 是否为空文件
- 是否已有 frontmatter

如果：
为空且无 frontmatter：

插入 metadata 模板。

----------------------------------------
2. 文件保存监听
----------------------------------------

监听文件修改事件。

如果：

文件包含：

updated:

字段。

则自动更新 updated 日期。

========================================
九、Frontmatter 处理要求
========================================

必须正确处理 YAML frontmatter。

推荐使用：

Obsidian 官方 API：

processFrontMatter()

或者安全 YAML 操作方式。

----------------------------------------

禁止：

使用正则表达式暴力替换整个 frontmatter。

----------------------------------------

必须：

仅修改目标字段。

========================================
十、兼容性要求
========================================

插件必须兼容：

- Obsidian Properties
- YAML Frontmatter
- Dataview
- Bases

========================================
十一、插件技术要求
========================================

使用：

- TypeScript
- 官方 Obsidian Plugin API

----------------------------------------

项目结构：

src/
    main.ts
    settings.ts
    template.ts
    metadata.ts
    types.ts

----------------------------------------

各文件职责：

main.ts
插件入口。

settings.ts
插件设置界面。

template.ts
metadata 模板生成。

metadata.ts
frontmatter 安全读写。

types.ts
类型定义（包含 VocabEntry 接口和默认 vocabulary）。

========================================
十二、性能要求
========================================

插件必须：

1. 不全库扫描
2. 不后台索引
3. 不监听过多事件
4. 不影响启动速度
5. 不增加大型依赖

========================================
十三、代码风格要求
========================================

要求：

- 类型安全
- 模块化
- 可维护
- 清晰注释
- 尽量避免复杂抽象
- 不过度工程化

========================================
十四、明确禁止实现的功能
========================================

不要实现：

- AI 自动分类
- 自动 domain 推断
- 自动 tags
- 自动摘要
- 历史文件批处理
- 智能知识分析
- 向量数据库
- embedding
- graph analysis
- OCR
- 网络请求
- 云同步

========================================
十五、最终设计目标
========================================

这个插件的目标不是"智能"。

而是：

构建一个：
- 极简
- 稳定
- 长期可维护
- 几乎无感存在

的 metadata 基础设施插件。

========================================
十六、输出要求
========================================

请直接生成：

1. 完整插件代码
2. package.json
3. manifest.json
4. tsconfig.json
5. 所有 TypeScript 文件
6. 设置页面实现
7. README
8. 可直接 npm install / build 的项目结构

确保：

可以直接作为 Obsidian 社区插件开发项目运行。
```
